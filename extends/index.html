React 高阶组件 -- 继承的用法

继承

“是一个”经验准则

继承的一种应用场景

  • 如果已经有一个类实现了我们想要的大部分行为，那么我们可以继承该类，并添加缺少的行为。

继承的两个问题：

  • 如果滥用继承，会得到很深的、难以理解和导航的类层次结构。
  • 我们会得到不一致的数据模型，其中的类并不合理

例如：定义一个 Point （点）类，一个 Circle （圆形）类

从 Point 继承，得到一个 Circle 类，并在该类上添加一个 radius 属性。

这种定义让人感觉很奇怪。

class Point {
  x: number;
  y: number;
  
  constructor(x: number, y: number) {
    this.x = x;
    this.y = y;
  }
}

class Circle extends Point {
  radius: number;

  constructor(x: number, y: number, radius: number) {
    super(x, y);
    this.radius = radius;
  }
}

子类的实例在逻辑上是父类的一个实例吗？上个例子中，答案是否定的。

Circle 不是一个 Point。

**继承和“是一个”关系**：继承会在子类型与父类型之间建立“是一个”关系。如果基类是 Shape, 派生类是 Circle，关系就是“Circle 是一个 Shape”。这是继承在语义上的含义，可用来判断是否应该在两个类型之间应用继承。

建模层次

当数据模型包含不同层次时，应当考虑继承。

继承的最佳用途：

  • 随着在继承链中向下移动，我们不断添加更多数据和行为来细化类型。

组合

面向对象编程的一个著名原则是，只要可能，就优先选择组合而不是继承。接下来介绍组合。

继续看 Point 和 Circle 示例，我们可以使 Circle 成为 Point 的子类，但这样就不合理。

对他们改造一番。

class Shape {
  id: string;

  constructor(id: string) {
    this.id = id;
  }
}

class Point {
  x: number;
  y: number;

  constructor(x: number, y: number) {
    this.x = x;
    this.y = y;
  }
}

class Circle extends Shape {
  center: Point;
  radius: number;
  
  constructor(id: string, center: Point, radius: number) {
    super(id);
    this.center = center;
    this.radius = radius;
  }
}

“有一个”经验准则

我们不是从一个类型继承行为，而是定义一个该类型的属性。使用这种技术时，我们能够得到被包含类型的状态，只不过是使其成为类型的组成部分，而不是继承部分。

**组合和“有一个”关系**：组合在容器类型和被包含类型之间建立了一种“有一个”关系。如果类型是 Circle，被包含的类是 Point，那么他们的关系是“Circle 有一个 Point”（该 Point 定义了 Circle 的圆心）。这是组合在语义上的意义，可用于判断是否应该在两个类型之间使用组合。

组合的一个主要优点是，组件属性的所有状态（例如 Circle 的 center 属性的坐标）被封装在这些组件中（例如 Point 类型的 center 属性），所以类型变得整洁多了。

React 组合组件

组件可以在其输出中引用其他组件。

例如，我们可以创建一个可以多次渲染 Welcome 组件的 App 组件：

function Welcome(props) {
  return <h1>Hello, {props.name}</h1>;
}

function App() {
  return (
    <div>
      <Welcome name="Sara" />
      <Welcome name="Cahal" />
      <Welcome name="Edite" />
    </div>
  );
}

ReactDOM.render(
  <App />,
  document.getElementById('root')
);

上面的组件，是组合。

React 高阶组件

react 官网

高阶组件（HOC）是 React 中用于复用组件逻辑的一种高级技巧。HOC 自身不是 React API 的一部分，它是一种基于 React 的组合特性而形成的设计模式。

具体而言，高阶组件是参数为组件，返回值为新组件的函数。

const EnhancedComponent = higherOrderComponent(WrappedComponent);

官方例子：

// 此函数接收一个组件...
function withSubscription(WrappedComponent, selectData) {
  // ...并返回另一个组件...
  return class extends React.Component {
    constructor(props) {
      super(props);
      this.handleChange = this.handleChange.bind(this);
      this.state = {
        data: selectData(DataSource, props)
      };
    }

    componentDidMount() {
      // ...负责订阅相关的操作...
      DataSource.addChangeListener(this.handleChange);
    }

    componentWillUnmount() {
      DataSource.removeChangeListener(this.handleChange);
    }

    handleChange() {
      this.setState({
        data: selectData(DataSource, this.props)
      });
    }

    render() {
      // ... 并使用新数据渲染被包装的组件!
      // 请注意，我们可能还会传递其他属性
      return <WrappedComponent data={this.state.data} {...this.props} />;
    }
  };
}

上面的例子，封装了初始化时，订阅外部数据源，移除组件时，取消订阅外部数据源的逻辑。

使同一套逻辑，多次复用。

上面的例子，还接受第二个参数 selecData，是一个处理数据机制，针对不同场景可以使用不同的数据处理方法，更加灵活。

上面的例子，是组合

只是组合的从属关系变了。被封装的 WrappedComponent，成为了 withSubscription 高阶组件的一部分。

React 高阶组件 反向继承

如果只是对原组件做封装，不需要知道原组件内部的状态，那用上面的例子就可以了。

但如果需要依赖原组件的内部状态呢？需要改变原组件的生命周期？就需要使用继承。

反向继承的作用：劫持渲染、获取并操作状态、改写生命周期或其他方法

例子：一个可以添加导航的高阶组件。

// 此函数接收一个组件...
function withNavigation(WrappedComponent) {
  // ...并返回另一个组件...
  return class extends WrappedComponent {
    constructor(props) {
      super(props);
    }

    componentDidMount() {
      super.componentDidMount && super.componentDidMount();
    }

    render() {
      return (
        <>
          <div>{super.title}</div>
          {super.render()}
        </>
      );
    }
  };
}

// 首页
class Home extends React.Componet {
  title = '首页'

  render() {
    return <div>我是首页</div>
  }
}

// 高阶组件的使用
withNavigation(Home)

继承后，使用 super 获取父类的状态和方法。

用“是一个”检验一下：带有导航的首页 是一个 首页。没问题。

在这里，反向继承做到了扩展功能的作用。锦上添花。

React 正向继承

如果你写的组件，是个基础组件，通过加入一些方法、状态，便可实现对应具体的功能, 那应该使用正向基础。

例如：你已经定义好了基类 BinaryExpression，通过继承扩展，分别实现了 加法表达式 SumExpression、乘法表达式 MulExpression；

interface IExpression {
  eval(): number;
}

// 表达式
abstract class BinaryExpression implements IExpression {
  constructor(a: number, b: number) {
    this.a = a;
    this.b = b;
  }
  abstract eval(): number;
}

// 加法表达式
class SumExpression extends BinaryExpression {
  eval(): number {
    return this.a + this.b;
  }
}

// 乘法表达式
class MulExpression extends BinaryExpression {
  eval(): number {
    return this.a * this.b;
  }
}

总结

反向继承：

  • 锦上添花。原组件已实现基本功能，此操作是在往上面，添加额外功能。

正向继承：

  • 基类提供了公共方法、逻辑框架，子类可在其基础上扩展，实现差异化功能；

如果想确定用正向继承，还是反向继承，可以用“是一个”来判断。

我的实践

最近写了个项目，有个订单详情页，三个以上的业务线会用到，功能、设计等基本一致。领导让做个封装，等后期增加业务线时，工作量可以减少。

思路：

  1. 页面封装，结构上分两种，通过访问链接看，更直观：
  • 通过
    https://dev.jd.com/a-order-detail、
    https://dev.jd.com/b-order-detail、
    https://dev.jd.com/c-order-detail 来访问不同业务线的订单详情。
  • 通过
    https://dev.jd.com/order-detail?type=a、
    https://dev.jd.com/order-detail?type=b、
    https://dev.jd.com/order-detail?type=c 来访问不同业务线的订单详情。

第一种，三条业务线的页面，是独立的。是物理意义上分开的三个页面。
第二种，三条业务线的页面，是聚合在一起的。通过 url 中参数，内部判断展示逻辑。

领导说用第1种。

  1. 封装。封装到什么程度呢？新增业务线时，需要写的代码越少越好。
  • 封装组件：页面设计稿基本一致，可以将页面拆成几个模块，分别封装成组件。
  • 封装逻辑：页面逻辑也基本一致，使用同一接口，获取订单详情，再渲染。
  1. 差异化。
  • 封装，是将共同点封装起来。不需要写重复代码。
  • 差异化，不同业务线必然是有差异化的，需要设计出尽量简单的对外接口。
